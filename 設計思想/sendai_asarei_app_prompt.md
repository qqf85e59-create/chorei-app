# 仙台事務所　朝礼運営アプリ　開発プロンプト

## 1. プロジェクト概要

### 1.1 目的
アタックス・ビジネス・コンサルティング仙台事務所のメンバー8名による朝礼を支援する運営アプリを開発する。関係性の質向上を目的とした朝礼の運営効率化、輪番管理、出欠記録、主題管理を行う。

### 1.2 対象ユーザー
- 運営担当（管理者）：1名
- 参加者（一般ユーザー）：8名
- 合計8名（運営担当は参加者を兼務する場合あり）

### 1.3 使用環境
- Webアプリケーション
- PC・スマートフォン両対応
- ブラウザから利用

### 1.4 開発スコープ（MVP）
第一フェーズ（2か月、週3回×8週＝24回）の運営に必要な機能を実装する。第二フェーズ以降の機能は運用中に段階的に追加する。

## 3. ユーザー区分と権限

### 3.1 運営（管理者・1名）
- 全機能の閲覧・編集権限

## 4. 画面仕様

### 4.1 起動時画面（初回・毎回）
アプリを開いた際、最初にグランドルールを表示する。

**表示内容**
```
朝礼について

この朝礼は、仙台事務所のメンバー同士の「関係性の質」を高める場です。
お互いをよく知り、率直に話せる関係をつくっていきたいと考えています。
参加にあたって、以下を共通のルールとします。

話す人は、テーマの範囲内で、何をどこまで話すかを自分で決めてかまいません。

聴く人は、話された内容について、その場で評価や感想を述べません。
反応の仕方はフェーズごとに運営から案内する形式に沿ってください。

運営は、進行の判断について質問された場合、どのルールに基づく判断かを説明します。

出欠は、参加を原則とします。休むとき・途中で抜けるときは運営に連絡してください。
理由は問いません。
```

- 「確認しました」ボタンで次画面へ進む
- 初回起動時は必須表示、以降は「グランドルール」メニューからいつでも参照可能

### 4.2 運営ダッシュボード
運営のみがアクセス可能なトップ画面。

**表示項目**
- ヘッダー：アタックスロゴ、「仙台事務所 朝礼運営」の文字
- 本日の朝礼情報
  - 日付、時刻
  - 主題
  - 発話者（名前、等級）
  - 応答者（第二フェーズのみ、現在は非表示）
- 本日の出欠状況
  - 出席者リスト
  - 欠席申告リスト
  - 未申告者リスト
- アラート領域
  - 連続欠席アラート（3回連続欠席者の表示）
- クイックアクセスボタン
  - 輪番計画
  - 主題カレンダー
  - 参加者マスタ
  - フェーズ進捗

### 4.3 参加者トップ画面
一般ユーザーがアクセスするトップ画面。

**表示項目**
- ヘッダー：アタックスロゴ、「仙台事務所 朝礼」の文字、ユーザー名
- 本日の朝礼情報
  - 日付、時刻
  - 主題
  - 発話者（名前）
- 自分の次の発話予定
  - 日付
  - 主題
- 自分の過去の発話履歴
  - 直近3回の発話（日付、主題）
- ボタン
  - 欠席・途中退出の申告
  - グランドルールを見る
  - 主題カレンダーを見る

### 4.4 カレンダー画面（全ユーザー閲覧可）
週単位・月単位で朝礼の予定を閲覧できる。

**表示項目**
- カレンダー形式（月表示・週表示切替可）
- 各日に以下を表示
  - 日付（祝日は別色で表示、朝礼非実施）
  - その日の発話者（名前、等級は運営のみ）
  - その週の主題
- フィルタ
  - 自分の発話予定のみ表示
  - 特定の主題で絞り込み
- クリックで詳細表示

**祝日スキップ機能**
- 日本の祝日は自動で朝礼非実施として表示
- 運営は祝日スキップ設定を手動上書き可能
- 祝日対応ライブラリを使用（例：japanese-holidays-js等）

### 4.5 輪番計画画面（運営のみ）
3か月分の朝礼の発話者・応答者・主題を管理する。

**表示項目**
- 一覧形式（日付、発話者、応答者、主題、ステータス）
- 行ごとに編集ボタン
- フィルタ（フェーズ、発話者、主題）
- 自動生成ボタン
  - 第一フェーズ：1巡目は「下位から上位」で自動生成、2・3巡目はランダム
  - 手動編集可能

**自動提案ロジック**
- 発話回数の自動均衡（誰が何回発話したかを追跡）
- ペアリング履歴の最適化（第二フェーズ用、実装は第二フェーズ移行時）
- 等級バランスチェック（第二フェーズ用）
- 欠席時の自動リスケジュール
  - 1巡目の欠席は後送り、順序を詰める
  - 1巡目冒頭の等級2（E2a・E2b）は例外（運営が手動判断）
- 連続欠席の段階的アラート
  - 2回連続：黄色警告
  - 3回連続：赤色警告＋運営への接触推奨
  - 4回以上：継続的な赤色警告

### 4.6 主題カレンダー画面（全ユーザー閲覧可）
8週間分の主題一覧を表示する。

**表示項目**
- 週番号、期間、主題
- 事前通知機能（週初に主題を全員に通知）
- 過去の主題履歴検索（第二フェーズ以降の主題も検索可能）

### 4.7 フェーズ進捗画面（全ユーザー閲覧可）
3フェーズ全体のロードマップと現在地を表示。

**表示項目**
- 第一フェーズ：個人理解期（5月〜6月）
- 第二フェーズ：中間接続期（7月〜9月）
- 第三フェーズ：議論期（10月〜）
- 現在のフェーズを強調表示
- フェーズごとの目的・形式の概要
- 進捗バー（週単位）

### 4.8 参加者マスタ画面（運営のみ）
参加者の情報を管理する。

**表示項目**
- ID、名前、等級、メールアドレス（任意）、権限区分
- 編集・追加・削除ボタン

### 4.9 欠席申告画面（全ユーザー）
欠席・途中退出の申告を行う。

**表示項目**
- 申告対象日の選択
- 区分選択（欠席・途中退出）
- 理由入力欄は設けない（グランドルール第4条との整合）
- 送信ボタン

### 4.10 グランドルール画面（全ユーザー）
グランドルールの全文を表示する。起動時画面と同じ内容を、いつでも参照可能な形で表示。

## 6. ビジネスロジック

### 6.1 輪番自動生成ロジック（第一フェーズ1巡目）
- 参加者を等級順にソートする（E2a, E2b, E3a, E3b, E3c, E4, E4p, E5）
- 開始日から週3回（火・木・金）の日程に、上記順序で割り当てる
- 祝日は自動スキップし、翌実施日に繰り延べる
- 9回目（1巡目終了翌日）は「運営内棚卸し」としてマークし、発話者を割り当てない

### 6.2 輪番自動生成ロジック（第一フェーズ2・3巡目）
- 参加者リストからランダムに順序を決定
- 運営が手動で調整可能
- 決定後、対象者に前日朝までに通知（通知機能はMVP範囲外、手動通知で対応）

### 6.3 欠席時の輪番調整
- 1巡目での欠席
  - 基本：欠席者を1巡目の最後尾に後送り、順序を詰める
  - 例外：1巡目冒頭のE2a・E2bの欠席は運営が手動判断（「下位から上位」の反転効果を守るため）
- 2巡目・3巡目の欠席
  - 運営が手動で調整

### 6.4 連続欠席検知
- 各参加者の直近の実施における出欠を追跡
- 連続欠席カウントを管理
- アラート段階
  - 2回連続：ダッシュボードに黄色警告
  - 3回連続：赤色警告＋運営への接触推奨表示
  - 4回以上：継続的な赤色警告

### 6.5 発話回数の均衡追跡
- 各参加者の発話回数を累計で管理
- 次の発話者候補を提案する際、発話回数が少ない人を優先
- 運営は提案を採用・変更可能

### 6.6 ペアリングロジック（第二フェーズ、実装は移行時）
- 発話者と応答者の等級差が2以内かチェック
- E5とE2の組み合わせは警告（応答者にE5を置く場合のみ許容、E5がE2の応答者になる組み合わせは禁止）
- 過去のペアリング履歴を参照し、同じ組み合わせが繰り返されないよう提案
- 3か月で全員が最低3〜4人と異なる応答者を経験するよう調整

### 6.7 祝日スキップロジック
- 日本の祝日（内閣府公表の祝日一覧に準拠）を自動判定
- 祝日は朝礼を実施しない
- 祝日で発話予定がずれた場合、翌実施日に繰り延べる
- 運営は個別に祝日スキップ設定を上書き可能

## 7. セキュリティ・プライバシー要件

### 7.1 認証・認可
- ユーザーID・パスワードによるログイン
- 運営と参加者の権限を厳密に分離
- セッション管理（ログアウト機能）

### 7.2 データアクセス制御
- 参加者は他者の出欠記録を閲覧不可
- 参加者は自分の発話履歴・出席記録のみ閲覧可
- 運営のみが全体の出欠記録・輪番計画を閲覧可

### 7.3 データの取り扱い
- 発話内容の記録は実装しない
- 参加者のパーソナルデータ分析機能は実装しない
- 出欠記録は朝礼運営目的以外に使用しない
- 欠席理由の必須入力は行わない

### 7.4 データ保存期間
- 実施記録：各フェーズ終了後3か月まで保管
- 以降は運営判断で保管継続または削除
- 削除機能を運営に提供

### 7.5 外部連携
- 外部システム（人事・勤怠・Slack・カレンダー等）との連携は実装しない
- アプリ内で完結する設計

1. 発話内容の記録（テキスト・音声・要約いずれも）

4. 参加者間のメッセージ機能
5. 外部システム連携（人事・勤怠・Slack・カレンダー）
6. 欠席理由の必須入力
7. 参加者全員への出欠記録公開

## 9. 技術スタック指定

### 9.1 推奨スタック
- フロントエンド：Next.js（React）+ TypeScript
- バックエンド：Next.js API Routes（または別途Node.js/Express）
- データベース：SQLite（ローカル開発）、本番はPostgreSQL
- 認証：NextAuth.js または独自実装
- UIライブラリ：Tailwind CSS + shadcn/ui
- 祝日判定：japanese-holidays-js または類似ライブラリ

### 9.2 代替スタック
上記スタックが不適な場合、以下も可。
- Streamlit（Python）：シンプルな実装を優先する場合
- Flask + React：軽量な構成を優先する場合

### 9.3 デプロイ
- 開発時：ローカル環境
- 本番：Vercel、または社内サーバー
- データベース：小規模利用のためSQLiteまたは軽量なPostgreSQL

## 12. 参考情報

### 12.1 第一フェーズの主題案（8週分）
- 第1週：最近気になっている景色や場所
- 第2週：休日に時間を使っている対象
- 第3週：最近ちょっといいなと思ったもの
- 第4週：子どもの頃に夢中だったこと
- 第5週：最近小さく困っていること
- 第6週：自分がよく行く店
- 第7週：最近考えている問い（業務外）
